## 介绍

经典计算机科学强调高效算法，追求尽可能快地完成计算。但许多联网程序花费时间的地方并不是计算，而是维持大量缓慢或很少有事件发生的连接。这类程序面临的是完全不同的挑战：怎样高效地等待海量网络事件。解决这个问题的一种现代方法，就是异步 I/O，也就是“async”。

本章将实现一个简单的 Web 爬虫。爬虫是异步应用的典型代表：它需要等待大量响应，却几乎不做什么计算。它一次能抓取的页面越多，整体完成得就越快。如果给每一个进行中的请求都分配一个线程，那么随着并发请求数量增长，它会在耗尽套接字之前先耗尽内存或其他线程相关资源。异步 I/O 则避免了这种对线程的依赖。

我们会分三个阶段来展示这个例子。首先，我们实现一个异步事件循环，并勾勒出一个基于回调的爬虫：它很高效，但一旦扩展到更复杂的问题，就会演变成难以维护的“意大利面条式”代码。其次，我们展示 Python 协程如何同时兼顾效率与可扩展性，并用生成器函数来实现简单协程。最后，我们将使用 Python 标准库 asyncio 中的完整协程机制，并借助异步队列来协调它们（见注 1）。

## 任务

一个 Web 爬虫会找出并下载某个网站上的所有页面，可能是为了归档，也可能是为了建立索引。它从一个根 URL 开始，抓取页面，解析其中指向尚未见过页面的链接，并把这些链接加入队列。当它抓到某个页面时，如果其中已经没有未访问过的链接，且队列也为空，那么它就停止。

我们可以通过并发下载多个页面来加快这个过程。爬虫一旦发现新链接，就会在不同的套接字上同时发起抓取。它在响应到达时进行解析，并把新发现的链接继续放入队列。当然，并发并不是越大越好，过高的并发可能会让性能开始下降，因此我们会限制并发请求数量，把剩余链接留在队列中，等一些进行中的请求结束后再处理。

## 传统的实现方法

怎样让爬虫具备并发能力？传统方法是创建线程池。每个线程负责通过一个套接字一次下载一个页面。比如，下载 `xkcd.com` 上的一个页面：

```
def fetch(url):
    sock = socket.socket()
    sock.connect(('xkcd.com', 80))
    request = 'GET {} HTTP/1.0\r\nHost: xkcd.com\r\n\r\n'.format(url)
    sock.send(request.encode('ascii'))
    response = b''
    chunk = sock.recv(4096)
    while chunk:
        response += chunk
        chunk = sock.recv(4096)

    # Page is now downloaded.
    links = parse_links(response)
    q.add(links)
```

默认情况下，套接字操作是阻塞的：当线程调用 `connect` 或 `recv` 这样的方法时，它会一直停在那里，直到操作完成（见注 2）。因此，要一次下载很多页面，我们就需要很多线程。成熟一些的应用会通过线程池保存空闲线程，把线程创建的成本分摊掉，再在后续任务中重复利用；对于套接字，也会类似地使用连接池。


然而，线程的开销是相对昂贵的，而且操作系统会对一个进程、一个用户乃至整台机器可拥有的线程数量施加各种硬限制。
在 Jesse 的电脑上，一个 Python 线程大约消耗 50KB 内存，而启动成千上万个线程就会失败。
如果我们把并发 Socket 操作扩展到数万级别，那么往往还没耗尽 Socket，就已经先把线程耗尽了。每线程的开销和系统对线程的限制，正是这种实现方式的瓶颈。


在那篇极具影响力的文章《The C10K problem》中，Dan Kegel 总结了多线程在 I/O 并发方面的局限性（见注 3）。他开篇写道：

> 是时候让web服务器同时处理数万客户端请求了，不是吗？毕竟，web那么大。

Kegel 在 1999 年提出了 “C10K” 这个术语。今天看来，一万个连接似乎已经不算什么，但问题本质上只是规模变了，类型并没有变。当年，对 C10K 采用“一个连接一个线程”的方式并不现实。如今这个上限提高了好几个数量级。事实上，我们这个玩具级爬虫用线程也完全能运行得很好；但对拥有数十万连接的大型应用来说，瓶颈依然存在：大多数系统仍然会在“还能继续创建套接字”之前，就已经“不能继续创建线程”了。那我们该如何突破这一点？


## 异步

异步 I/O 框架使用**非阻塞**套接字，在单线程上并发执行操作。在我们的异步爬虫中，开始与服务器建立连接之前，会先把套接字设为非阻塞：

```python
sock = socket.socket()
sock.setblocking(False)
try:
    sock.connect(('xkcd.com', 80))
except BlockingIOError:
    pass
```

让人头疼的是，即使一切正常，非阻塞套接字在调用 `connect` 时也会抛出异常。这其实只是底层 C 函数令人头疼的行为在 Python 层的延续：它会把 `errno` 设为 `EINPROGRESS`，用来告诉你连接已经开始了。

现在，爬虫还需要一种方式来判断连接何时建立，这样它才能发送 HTTP 请求。我们当然可以在一个紧密循环里反复尝试：

```python
request = 'GET {} HTTP/1.0\\r\\nHost: xkcd.com\\r\\n\\r\\n'.format(url)
encoded = request.encode('ascii')

while True:
    try:
        sock.send(encoded)
        break  # Done.
    except OSError as e:
        pass

print('sent')
```

这种方法不仅浪费电，而且也无法高效地同时等待**多个**套接字上的事件。很久以前，BSD Unix 用 `select` 这个 C 函数来解决这个问题：它能等待一个非阻塞套接字，或者一小组这类套接字上的事件发生。如今，随着互联网应用对海量连接的需求不断增长，又出现了 `poll`、BSD 上的 `kqueue` 以及 Linux 上的 `epoll` 等替代方案。这些 API 与 `select` 类似，但在非常大的连接规模下性能更好。

Python 3.4 的 `DefaultSelector` 会在当前系统上选择最合适的 `select` 风格函数。为了接收网络 I/O 的通知，我们创建一个非阻塞套接字，并把它注册到默认 selector：

```python
from selectors import DefaultSelector, EVENT_WRITE

selector = DefaultSelector()

sock = socket.socket()
sock.setblocking(False)
try:
    sock.connect(('xkcd.com', 80))
except BlockingIOError:
    pass

def connected():
    selector.unregister(sock.fileno())
    print('connected!')

selector.register(sock.fileno(), EVENT_WRITE, connected)
```

我们忽略那次伪错误，然后调用 `selector.register`，传入套接字的文件描述符，以及一个表示“我们在等待什么事件”的常量。为了在连接建立时收到通知，我们传入 `EVENT_WRITE`：也就是说，我们想知道套接字何时“可写”。同时还传入一个 Python 函数 `connected`，让它在事件发生时运行。这样的函数就叫做**回调**（callback）。

我们在循环中处理 selector 收到的 I/O 通知：

```python
def loop():
    while True:
        events = selector.select()
        for event_key, event_mask in events:
            callback = event_key.data
            callback()
```

一旦非阻塞套接字连接成功，`connected` 回调就保存在 `event_key.data` 中，我们把它取出来并执行。

和上面那个疯狂空转的循环不同，这里的 `select` 调用会阻塞等待下一个 I/O 事件。随后，事件循环运行等待这些事件的回调。尚未完成的操作会继续保持挂起状态，等待未来某个事件循环 tick 再继续。

到这里我们已经演示了什么？我们展示了如何启动一个操作，并在操作就绪时执行回调。异步**框架**正是建立在我们刚刚看到的两个特性之上——非阻塞套接字与事件循环——来在单线程上运行并发操作。

这里实现的是“并发”，但并不是传统意义上的“并行”。也就是说，我们构建的是一个 I/O 重叠的小系统：它能在一些操作尚未完成时启动新的操作，但并没有真正利用多个 CPU 核心并行执行计算。不过，这个系统本来就是为 I/O 密集型问题设计的，而不是 CPU 密集型问题（见注 4）。

因此，我们的事件循环在并发 I/O 上之所以高效，是因为它不会把线程资源分配给每个连接。但在继续之前，有必要纠正一个常见误解：异步不一定比多线程**更快**。很多时候并非如此——事实上，在 Python 中，像我们这样的事件循环在处理少量但非常活跃的连接时，往往会比多线程略慢一些。如果没有全局解释器锁，线程在这类负载下甚至会表现得更好。异步 I/O 真正适合的，是那些拥有大量缓慢或“沉睡”连接、并且事件并不频繁的应用（见注 5）。

## 回调

在我们目前搭好的这个简陋异步框架之上，应该如何构建一个 Web 爬虫？哪怕只是写一个简单的 URL 获取器，过程都相当痛苦。

先定义两个全局集合：一个保存还没抓取的 URL，另一个保存已经见过的 URL：

```python
urls_todo = set(['/'])
seen_urls = set(['/'])
```

`seen_urls` 包含了 `urls_todo` 里的 URL 以及已经完成抓取的 URL。这两个集合都以根路径 `"/"` 作为初始值。

抓取一个页面需要一连串回调。套接字连接成功时，`connected` 回调会被触发，并向服务器发送 GET 请求。但随后它还要等待响应，于是又得注册另一个回调。如果那个回调触发时还无法读到完整响应，就再次注册自己，如此反复。

让我们把这些回调收拢到一个 `Fetcher` 对象中。它需要一个 URL、一个套接字对象，以及一个用来累计响应字节的地方：

```python
class Fetcher:
    def __init__(self, url):
        self.response = b''  # Empty array of bytes.
        self.url = url
        self.sock = None
```

我们首先调用 `Fetcher.fetch`：

```python
    # Method on Fetcher class.
    def fetch(self):
        self.sock = socket.socket()
        self.sock.setblocking(False)
        try:
            self.sock.connect(('xkcd.com', 80))
        except BlockingIOError:
            pass

        # Register next callback.
        selector.register(self.sock.fileno(),
                          EVENT_WRITE,
                          self.connected)
```

`fetch` 方法启动了套接字连接。但请注意：在连接真正建立之前，这个方法就已经返回了。它必须把控制权交回事件循环，让事件循环去等待连接完成。为什么？假设我们的整个程序是这样组织的：

```python
# Begin fetching http://xkcd.com/353/
fetcher = Fetcher('/353/')
fetcher.fetch()

while True:
    events = selector.select()
    for event_key, event_mask in events:
        callback = event_key.data
        callback(event_key, event_mask)
```

所有事件通知都在事件循环调用 `select` 时处理。因此，`fetch` 必须把控制权交还给事件循环，这样程序才知道套接字何时连上。也只有到那时，循环才会执行它在 `fetch` 末尾注册好的 `connected` 回调。

`connected` 的实现如下：

```python
    # Method on Fetcher class.
    def connected(self, key, mask):
        print('connected!')
        selector.unregister(key.fd)
        request = 'GET {} HTTP/1.0\\r\\nHost: xkcd.com\\r\\n\\r\\n'.format(self.url)
        self.sock.send(request.encode('ascii'))

        # Register the next callback.
        selector.register(key.fd,
                          EVENT_READ,
                          self.read_response)
```

这个方法发送一个 GET 请求。真正的生产级程序会检查 `send` 的返回值，以防整条消息无法一次发完。但我们的请求很小，程序也很简单，于是直接调用 `send`，然后开始等待响应。当然，它还必须注册另一个回调，并再次把控制权交给事件循环。下一个也是最后一个回调 `read_response` 用来处理服务器返回的内容：

```python
    # Method on Fetcher class.
    def read_response(self, key, mask):
        global stopped

        chunk = self.sock.recv(4096)  # 4k chunk size.
        if chunk:
            self.response += chunk
        else:
            selector.unregister(key.fd)  # Done reading.
            links = self.parse_links()

            # Python set-logic:
            for link in links.difference(seen_urls):
                urls_todo.add(link)
                Fetcher(link).fetch()  # <- New Fetcher.

            seen_urls.update(links)
            urls_todo.remove(self.url)
            if not urls_todo:
                stopped = True
```

每当 selector 发现套接字“可读”时，这个回调都会被执行。而“可读”可能意味着两件事：套接字里有数据，或者套接字已经关闭。

这个回调会尝试从套接字中读取最多 4KB 数据。如果当前可读数据不足 4KB，`chunk` 就只包含现有的数据；如果数据更多，`chunk` 就会正好是 4KB，而套接字仍会保持可读，于是事件循环会在下一次 tick 中再次运行这个回调。等整个响应读取完毕时，服务器已经关闭了套接字，这时 `chunk` 就为空。

这里没有展示的 `parse_links` 方法会返回一个 URL 集合。我们会为每一个新 URL 启动一个新的 fetcher，而且这里没有任何并发上限。回调式异步编程有个不错的地方：当我们修改像 `seen_urls` 这样的共享数据时，不需要加互斥锁。因为这里没有抢占式多任务调度，我们不会在代码的任意位置被打断。

再增加一个全局变量 `stopped` 来控制循环：

```python
stopped = False

def loop():
    while not stopped:
        events = selector.select()
        for event_key, event_mask in events:
            callback = event_key.data
            callback()
```

所有页面下载完后，fetcher 会让全局事件循环停止，程序也就退出了。

这个例子把异步的一大问题暴露得非常直接：意大利面条式代码。我们需要一种方式来表达一系列计算与 I/O 操作，并让多条这样的操作链并发运行。但在没有线程的情况下，一系列操作无法自然地收拢到同一个函数里：每次函数发起 I/O 操作时，都必须显式保存未来还会用到的状态，然后返回。如何保存这些状态，完全要靠程序员自己考虑并手写实现。

我们具体说明一下。想一想用传统阻塞套接字在线程里抓取一个 URL 时，代码写起来多么简单：

```python
# Blocking version.
def fetch(url):
    sock = socket.socket()
    sock.connect(('xkcd.com', 80))
    request = 'GET {} HTTP/1.0\\r\\nHost: xkcd.com\\r\\n\\r\\n'.format(url)
    sock.send(request.encode('ascii'))
    response = b''
    chunk = sock.recv(4096)
    while chunk:
        response += chunk
        chunk = sock.recv(4096)

    # Page is now downloaded.
    links = parse_links(response)
    q.add(links)
```

这个函数在一次套接字操作和下一次套接字操作之间记住了哪些状态？它记住了套接字、URL，以及不断累积的 `response`。一个在线程上运行的函数，可以依赖编程语言本身的基本能力，把这些临时状态保存在局部变量中，也就是保存在它的栈上。这个函数还有一个“延续”（continuation），也就是 I/O 完成后它打算继续执行的代码。运行时通过保存线程的指令指针来记住这段延续。I/O 完成之后，如何恢复局部变量与延续，这些都已经由语言机制内建完成了。

但在基于回调的异步框架中，这些语言特性帮不上忙。在等待 I/O 时，函数必须显式保存自己的状态，因为函数会在 I/O 完成之前返回，并丢失它的栈帧。为了替代局部变量，我们在基于回调的示例里把 `sock` 和 `response` 存成 `self`（也就是 Fetcher 实例）的属性；为了替代指令指针，我们通过注册 `connected` 与 `read_response` 这两个回调来保存“接下来要做什么”。随着应用功能增长，跨回调手工保存的状态会越来越复杂，这种繁琐的簿记工作足以让程序员偏头痛发作。

更糟的是，如果某个回调在安排链条中的下一个回调之前就抛出了异常，会发生什么？假设我们的 `parse_links` 方法写得不好，在解析某段 HTML 时抛出了异常：

```
Traceback (most recent call last):
  File "loop-with-callbacks.py", line 111, in <module>
    loop()
  File "loop-with-callbacks.py", line 106, in loop
    callback(event_key, event_mask)
  File "loop-with-callbacks.py", line 51, in read_response
    links = self.parse_links()
  File "loop-with-callbacks.py", line 67, in parse_links
    raise Exception('parse error')
Exception: parse error
```

这段栈追踪只说明了事件循环正在运行一个回调。我们根本看不出错误是如何一步步走到这里的。调用链两头都断掉了：不知道自己原本要去哪里，也不知道自己从哪里来。这种上下文丢失被称为“栈撕裂”（stack ripping），在很多情况下都会让排查问题的人无从下手。栈撕裂还让我们无法像包裹普通函数调用树那样，用一个 `try / except` 块来包住整条回调链（见注 6）。

因此，抛开多线程与异步谁更高效的长期争论不谈，还有另一场争论：到底哪一种更容易出错？线程一旦同步写错，就容易出现数据竞争；而回调由于栈撕裂，又非常难调试。

## 协程

现在我们给出一个诱人的承诺：确实有办法写出一种异步代码，它既拥有回调式代码的高效率，又保留多线程程序那种经典而清爽的外观。这种组合就是通过“协程”实现的。使用 Python 3.4 标准库 asyncio 和一个名为 `aiohttp` 的包，在协程里抓取一个 URL 可以写得非常直接（见注 7）：

```python
    @asyncio.coroutine
    def fetch(self, url):
        response = yield from self.session.get(url)
        body = yield from response.read()
```

而且它也很容易扩展。相比“每个线程约 50KB 内存”以及操作系统对线程数量的硬限制，在 Jesse 的机器上，一个 Python 协程只占用不到 3KB 内存。Python 可以轻松启动几十万个协程。

协程这个概念可以追溯到计算机科学的早期。它本质上很简单：一种可以暂停，也可以恢复执行的子程序。线程由操作系统进行抢占式多任务调度；协程则采用协作式多任务调度：它们自己决定何时暂停，以及接下来轮到谁运行。

协程有很多种实现方式，即使只看 Python 也不止一种。Python 3.4 标准库 asyncio 中的协程建立在生成器、Future 类以及 `yield from` 语句之上。从 Python 3.5 开始，协程已经是语言本身的原生特性（见注 8）；不过，理解它在 Python 3.4 中如何借助既有语言机制实现，依然是掌握 Python 3.5 原生协程的基础。

为了说明 Python 3.4 基于生成器的协程，我们先花一点篇幅解释生成器，以及它们在 asyncio 中是如何被当作协程使用的。等这个铺垫完成之后，我们就会回到异步 Web 爬虫这个目标上来。

## Python生成器如何工作

在理解 Python 生成器之前，你得先明白普通 Python 函数是如何工作的。通常，当一个 Python 函数调用子程序时，控制权会一直留在子程序里，直到它返回或者抛出异常，然后控制权才回到调用者：

```python
>>> def foo():
...     bar()
...
>>> def bar():
...     pass
```

标准 Python 解释器是用 C 写的。执行 Python 函数的那个 C 函数，名字优雅地叫做 `PyEval_EvalFrameEx`。它接收一个 Python 栈帧对象，并在这个栈帧的上下文中解释执行 Python 字节码。下面是 `foo` 的字节码：

```python
>>> import dis
>>> dis.dis(foo)
  2           0 LOAD_GLOBAL              0 (bar)
              3 CALL_FUNCTION            0 (0 positional, 0 keyword pair)
              6 POP_TOP
              7 LOAD_CONST               0 (None)
             10 RETURN_VALUE
```

`foo` 会把 `bar` 压入栈并调用它，然后把返回值从栈里弹出，再把 `None` 压栈并返回 `None`。

当 `PyEval_EvalFrameEx` 遇到 `CALL_FUNCTION` 字节码时，它会创建一个新的 Python 栈帧并递归调用自己，用这个新栈帧去执行 `bar`。

理解下面这件事非常关键：Python 的栈帧对象是分配在堆内存上的！Python 解释器本身是一个普通的 C 程序，因此它自己的 C 栈帧当然在真正的调用栈上；但它操作的 **Python** 栈帧却在堆里。这意味着一个 Python 栈帧可以比它所属的函数调用活得更久。你可以在 `bar` 内部把当前帧保存起来，然后在交互式环境中观察：

```python
>>> import inspect
>>> frame = None
>>> def foo():
...     bar()
...
>>> def bar():
...     global frame
...     frame = inspect.currentframe()
...
>>> foo()
>>> # The frame was executing the code for 'bar'.
>>> frame.f_code.co_name
'bar'
>>> # Its back pointer refers to the frame for 'foo'.
>>> caller_frame = frame.f_back
>>> caller_frame.f_code.co_name
'foo'
```

到这里，Python 生成器登场所需的舞台已经搭好了：它们正是用这些相同的构件——代码对象与栈帧——发挥出奇妙效果。

下面是一个生成器函数：

```python
>>> def gen_fn():
...     result = yield 1
...     print('result of yield: {}'.format(result))
...     result2 = yield 2
...     print('result of 2nd yield: {}'.format(result2))
...     return 'done'
...
```

当 Python 把 `gen_fn` 编译成字节码时，它一看到 `yield` 语句，就知道这不是普通函数，而是一个生成器函数。于是它会设置一个标志位来记住这件事：

```python
>>> # The generator flag is bit position 5.
>>> generator_bit = 1 << 5
>>> bool(gen_fn.__code__.co_flags & generator_bit)
True
```

当你调用生成器函数时，Python 看见这个标志位，就不会真的执行函数体。相反，它会先创建一个生成器对象：

```python
>>> gen = gen_fn()
>>> type(gen)
<class 'generator'>
```

一个 Python 生成器封装了一个栈帧，以及对某段代码的引用，也就是 `gen_fn` 的函数体：

```python
>>> gen.gi_code.co_name
'gen_fn'
```

所有由 `gen_fn` 创建出来的生成器都指向同一份代码，但每个生成器都有自己的栈帧。这个栈帧并不在任何真实的调用栈上，而是待在堆内存中等待使用。

这个栈帧有一个“最后执行指令”指针，也就是它最近执行过的指令位置。刚开始时，这个指针是 -1，表示生成器尚未开始：

```python
>>> gen.gi_frame.f_lasti
-1
```

当我们调用 `send` 时，生成器运行到它的第一个 `yield` 并暂停。`send` 的返回值是 1，因为这正是 `gen` 交给 `yield` 表达式的值：

```python
>>> gen.send(None)
1
```

这时，生成器的指令指针距离开头已经前进了 3 个字节码，而整段编译后的 Python 字节码共有 56 个字节：

```python
>>> gen.gi_frame.f_lasti
3
>>> len(gen.gi_code.co_code)
56
```

生成器可以在任何时候、从任何函数中恢复执行，因为它的栈帧并不真的待在调用栈上，而是待在堆里。因此它在调用层级中的位置并不固定，也不必像普通函数那样遵守“后进先出”的执行顺序。它被解放出来了，像云一样漂浮着。

我们可以把值 `"hello"` 发送给生成器，这个值会成为 `yield` 表达式的结果，然后生成器继续运行，直到它再一次 `yield` 出 2：

```python
>>> gen.send('hello')
result of yield: hello
2
```

这时，它的栈帧里已经包含了局部变量 `result`：

```python
>>> gen.gi_frame.f_locals
{'result': 'hello'}
```

由 `gen_fn` 创建出的其他生成器，也会拥有各自独立的栈帧与局部变量。

当我们再次调用 `send` 时，生成器会从第二个 `yield` 继续执行，并最终通过抛出特殊的 `StopIteration` 异常结束：

```python
>>> gen.send('goodbye')
result of 2nd yield: goodbye
Traceback (most recent call last):
  File "<input>", line 1, in <module>
StopIteration: done
```

这个异常携带了一个值，也就是生成器的返回值：字符串 `"done"`。

## 使用生成器实现协程

所以，生成器可以暂停，可以带着一个值恢复执行，而且还拥有返回值。把它当成构建异步编程模型的原语，看起来非常合适，而且不必忍受意大利面条式的回调！我们想构建的是一种“协程”：一种会和程序中其他例程协作调度的例程。我们这里实现的协程，是 Python 标准库 asyncio 协程的简化版本。与 asyncio 一样，我们会使用生成器、Future，以及 `yield from` 语句。

首先我们需要一种方式来表示“协程正在等待的未来结果”。下面是一个精简版实现：

```python
class Future:
    def __init__(self):
        self.result = None
        self._callbacks = []

    def add_done_callback(self, fn):
        self._callbacks.append(fn)

    def set_result(self, result):
        self.result = result
        for fn in self._callbacks:
            fn(self)
```

一个 future 初始状态是“pending”。调用 `set_result` 之后，它就被“resolved”了（见注 9）。

现在，让我们把前面的 fetcher 改成使用 future 与协程。回忆一下之前用回调写的 `fetch`：

```python
class Fetcher:
    def fetch(self):
        self.sock = socket.socket()
        self.sock.setblocking(False)
        try:
            self.sock.connect(('xkcd.com', 80))
        except BlockingIOError:
            pass
        selector.register(self.sock.fileno(),
                          EVENT_WRITE,
                          self.connected)

    def connected(self, key, mask):
        print('connected!')
        # And so on....
```

`fetch` 方法先开始连接套接字，然后注册回调 `connected`，等套接字就绪时再执行。现在我们可以把这两步合并成一个协程：

```python
    def fetch(self):
        sock = socket.socket()
        sock.setblocking(False)
        try:
            sock.connect(('xkcd.com', 80))
        except BlockingIOError:
            pass

        f = Future()

        def on_connected():
            f.set_result(None)

        selector.register(sock.fileno(),
                          EVENT_WRITE,
                          on_connected)
        yield f
        selector.unregister(sock.fileno())
        print('connected!')
```

现在，`fetch` 已经不是普通函数，而是一个生成器函数，因为它包含了 `yield` 语句。我们先创建一个待完成的 future，再把它 `yield` 出去，让 `fetch` 暂停，直到套接字准备好。内部函数 `on_connected` 则负责把 future 解析完成。

但当 future 完成时，究竟是谁来恢复这个生成器？我们需要一个协程**驱动器**。姑且把它叫作 “task”：

```python
class Task:
    def __init__(self, coro):
        self.coro = coro
        f = Future()
        f.set_result(None)
        self.step(f)

    def step(self, future):
        try:
            next_future = self.coro.send(future.result)
        except StopIteration:
            return

        next_future.add_done_callback(self.step)

# Begin fetching http://xkcd.com/353/
fetcher = Fetcher('/353/')
Task(fetcher.fetch())

loop()
```

这个 task 通过把 `None` 发送进 `fetch` 生成器来启动它。随后，`fetch` 会一直运行，直到它 `yield` 出一个 future，这个 future 被 task 接收为 `next_future`。当套接字连接完成时，事件循环会执行回调 `on_connected`，它让 future 完成；而 future 完成后又会调用 `step`，从而恢复 `fetch` 的执行。

## 使用`yield from`构造协程

一旦套接字连上，我们就要发送 HTTP GET 请求并读取服务器响应。现在这些步骤已经不必再分散在多个回调里了；我们可以把它们收拢到同一个生成器函数中：

```python
    def fetch(self):
        # ... connection logic from above, then:
        sock.send(request.encode('ascii'))

        while True:
            f = Future()

            def on_readable():
                f.set_result(sock.recv(4096))

            selector.register(sock.fileno(),
                              EVENT_READ,
                              on_readable)
            chunk = yield f
            selector.unregister(sock.fileno())
            if chunk:
                self.response += chunk
            else:
                # Done reading.
                break
```

这段代码从套接字读取整条消息，看起来很有复用价值。那要怎样把它从 `fetch` 中拆成子程序呢？这时就轮到 Python 3 那个著名的 `yield from` 登场了。它允许一个生成器把工作**委托**给另一个生成器。

要理解它的机制，我们先回到前面那个简单的生成器示例：

```python
>>> def gen_fn():
...     result = yield 1
...     print('result of yield: {}'.format(result))
...     result2 = yield 2
...     print('result of 2nd yield: {}'.format(result2))
...     return 'done'
...
```

如果要在另一个生成器中调用它，可以借助 `yield from` 来委托：

```python
>>> # Generator function:
>>> def caller_fn():
...     gen = gen_fn()
...     rv = yield from gen
...     print('return value of yield-from: {}'
...           .format(rv))
...
>>> # Make a generator from the
>>> # generator function.
>>> caller = caller_fn()
```

`caller` 这个生成器的行为几乎就像它所委托的 `gen` 一样：

```python
>>> caller.send(None)
1
>>> caller.gi_frame.f_lasti
15
>>> caller.send('hello')
result of yield: hello
2
>>> caller.gi_frame.f_lasti  # Hasn't advanced.
15
>>> caller.send('goodbye')
result of 2nd yield: goodbye
return value of yield-from: done
Traceback (most recent call last):
  File "<input>", line 1, in <module>
StopIteration
```

当 `caller` 正在 `yield from gen` 时，它自己不会前进。注意，它的指令指针始终停留在 15，也就是 `yield from` 所在的位置，尽管内部的 `gen` 已经从一个 `yield` 前进到下一个 `yield` 了（见注 10）。从 `caller` 外部看，我们分辨不出它产出的值究竟来自 `caller` 自己，还是来自它所委托的生成器；而从 `gen` 内部看，我们也分辨不出传进来的值究竟是来自 `caller`，还是来自更外层调用者。`yield from` 就像一条无摩擦的通道，让值在 `gen` 完成之前自由地进出。

协程可以通过 `yield from` 把工作委托给子协程，并接收子协程返回的结果。注意上面 `caller` 打印了 `return value of yield-from: done`。当 `gen` 执行完毕时，它的返回值会变成 `caller` 中那条 `yield from` 语句的值：

```python
    rv = yield from gen
```

前面我们批评基于回调的异步编程时，最强烈的抱怨之一就是“栈撕裂”：一旦回调抛出异常，栈追踪往往没什么帮助。它只能告诉你事件循环正在运行某个回调，却不会告诉你**为什么**。协程在这方面表现如何呢？

```python
>>> def gen_fn():
...     raise Exception('my error')
>>> caller = caller_fn()
>>> caller.send(None)
Traceback (most recent call last):
  File "<input>", line 1, in <module>
  File "<input>", line 3, in caller_fn
  File "<input>", line 2, in gen_fn
Exception: my error
```

这就有用得多了！栈追踪明确展示出：`caller_fn` 在委托给 `gen_fn` 时，`gen_fn` 抛出了这个错误。更让人安心的是，我们还能像对普通子程序那样，用异常处理器包裹对子协程的调用：

```python
>>> def gen_fn():
...     yield 1
...     raise Exception('uh oh')
...
>>> def caller_fn():
...     try:
...         yield from gen_fn()
...     except Exception as exc:
...         print('caught {}'.format(exc))
...
>>> caller = caller_fn()
>>> caller.send(None)
1
>>> caller.send('hello')
caught uh oh
```

因此，我们可以像拆分普通子程序一样拆分协程逻辑。让我们从 fetcher 中抽出几个有用的子协程。先写一个 `read` 协程，用来读取一个块：

```python
def read(sock):
    f = Future()

    def on_readable():
        f.set_result(sock.recv(4096))

    selector.register(sock.fileno(), EVENT_READ, on_readable)
    chunk = yield f  # Read one chunk.
    selector.unregister(sock.fileno())
    return chunk
```

然后在 `read` 之上构建 `read_all` 协程，用来读取完整消息：

```python
def read_all(sock):
    response = []
    # Read whole response.
    chunk = yield from read(sock)
    while chunk:
        response.append(chunk)
        chunk = yield from read(sock)

    return b''.join(response)
```

如果你稍微眯起眼睛，让那些 `yield from` 语句变得模糊，这两个函数看起来就像执行阻塞 I/O 的普通函数。可事实上，`read` 和 `read_all` 都是协程。从 `read_all` 里 `yield from read`，会让 `read_all` 暂停，直到 I/O 完成；当 `read_all` 暂停时，asyncio 的事件循环会去做别的工作、等待别的 I/O；等对应事件就绪后，再在下一轮事件循环中用 `read` 的结果恢复 `read_all`。

在调用栈的最外层，`fetch` 调用 `read_all`：

```python
class Fetcher:
    def fetch(self):
         # ... connection logic from above, then:
        sock.send(request.encode('ascii'))
        self.response = yield from read_all(sock)
```

神奇的是，`Task` 类完全不需要修改。它和之前一样，照样驱动最外层的 `fetch` 协程：

```python
Task(fetcher.fetch())
loop()
```

当 `read` `yield` 出一个 future 时，这个 future 会沿着一连串 `yield from` 语句向上传递，就好像它是直接从 `fetch` 中 `yield` 出来的一样。当循环让这个 future 完成时，task 会把结果送进 `fetch`，而这个值又会被 `read` 接收到，就好像 task 直接驱动 `read` 一样。

为了让我们的协程实现更完善一些，还可以再抛光一个细节：目前我们的代码在等待 future 时用的是 `yield`，而在委托给子协程时用的是 `yield from`。如果无论何时协程暂停都统一使用 `yield from`，那就会更优雅一些。这样，协程本身就不必关心自己等待的到底是什么类型的对象。

这里可以利用 Python 中“生成器与迭代器深度对应”这一点。对调用者来说，推进一个生成器，与推进一个迭代器没有本质区别。因此，我们通过实现一个特殊方法，让 `Future` 也变得可迭代：

```python
    # Method on Future class.
    def __iter__(self):
        # Tell Task to resume me here.
        yield self
        return self.result
```

`Future` 的 `__iter__` 方法本身就是一个协程，它把 future 自己 `yield` 出去。现在，如果我们把下面这样的代码：

```python
# f is a Future.
yield f
```

替换成：

```python
# f is a Future.
yield from f
```

结果完全一样！驱动它的 task 仍然会在调用 `send` 时拿到这个 future，而等 future 完成之后，又会把新结果送回协程中。

为什么要处处使用 `yield from`？为什么这比“等待 future 时用 `yield`，委托给子协程时用 `yield from`”更好？因为这样一来，一个方法就可以自由改变自己的实现，而不会影响调用者：它既可能是一个普通方法，返回一个将来会**解析成某个值**的 future；也可能是一个内部带有 `yield from` 语句、最终**返回某个值**的协程。不管是哪种情况，调用者都只需要对这个方法 `yield from`，就可以等待结果。

亲爱的读者，关于 asyncio 协程的这段愉快讲解到这里就告一段落了。我们窥探了生成器的内部机制，勾勒出 futures 和 tasks 的实现方式，也说明了 asyncio 如何兼得两边的好处：它让并发 I/O 比线程更高效，也比回调更易读。当然，真正的 asyncio 比我们这个草图复杂得多。真实框架还要处理零拷贝 I/O、公平调度、异常处理，以及大量其他特性。

对于 asyncio 的使用者来说，用协程编程远比我们这里展示的过程简单。在上面的代码里，我们从第一原则出发手工实现了协程，所以你会看到回调、task、future，甚至非阻塞套接字与 `select` 调用。但当你真正使用 asyncio 构建应用时，这些实现细节都不会直接出现在你的代码里。正如我们承诺的那样，抓取一个 URL 可以写得非常利落：

```python
    @asyncio.coroutine
    def fetch(self, url):
        response = yield from self.session.get(url)
        body = yield from response.read()
```

现在，带着对这些底层机制的满足感，我们回到最初的任务：用 asyncio 写一个异步 Web 爬虫。

## 协调协程

一开始我们已经描述过理想中的爬虫应该如何工作。现在是时候用 asyncio 协程把它真正实现出来了。

我们的爬虫会先抓取第一页，解析其中的链接，并把它们加入队列。之后，它会在整个网站范围内扩散开来，并发抓取页面。但为了限制客户端与服务器的负载，我们希望最多只有固定数量的 worker 同时运行，不能再多。每当某个 worker 抓完一个页面，它就应该立刻从队列中取下一个链接。整个过程中，可能会出现工作量暂时不足的时段，这时一些 worker 必须暂停；而当某个 worker 遇到一个包含大量新链接的页面时，队列又会突然增长，之前暂停的 worker 就应该醒来继续干活。最后，当所有工作都做完后，程序必须退出。

如果这些 worker 是线程，我们会怎样表达这个算法？我们可以使用 Python 标准库中的同步队列（见注 11）。每次往队列里放一个项目时，队列都会增加“任务数”计数；worker 线程处理完一个项目后调用 `task_done`；主线程则阻塞在 `Queue.join` 上，直到队列里每一个放进去的项目都收到对应的一次 `task_done`，然后主线程退出。

协程使用 asyncio 队列时，模式完全一样！首先导入它（见注 12）：

```python
try:
    from asyncio import JoinableQueue as Queue
except ImportError:
    # In Python 3.5, asyncio.JoinableQueue is
    # merged into Queue.
    from asyncio import Queue
```

我们把 worker 共享的状态收拢进一个 crawler 类，并把主逻辑写在它的 `crawl` 方法里。然后把 `crawl` 当成协程启动，并让 asyncio 的事件循环一直运行到 `crawl` 完成：

```python
loop = asyncio.get_event_loop()

crawler = crawling.Crawler('http://xkcd.com',
                           max_redirect=10)

loop.run_until_complete(crawler.crawl())
```

这个爬虫初始化时会接收根 URL 和 `max_redirect`，也就是针对任意一个 URL 最多愿意跟随多少次重定向。它会把 `(URL, max_redirect)` 这个二元组放入队列。（至于为什么要这么做，稍后解释。）

```python
class Crawler:
    def __init__(self, root_url, max_redirect):
        self.max_tasks = 10
        self.max_redirect = max_redirect
        self.q = Queue()
        self.seen_urls = set()

        # aiohttp's ClientSession does connection pooling and
        # HTTP keep-alives for us.
        self.session = aiohttp.ClientSession(loop=loop)

        # Put (URL, max_redirect) in the queue.
        self.q.put((root_url, self.max_redirect))
```

这时，队列中未完成任务的数量是 1。回到主脚本后，我们启动事件循环以及 `crawl` 方法：

```python
loop.run_until_complete(crawler.crawl())
```

`crawl` 协程负责启动 worker。它很像一个主线程：它会阻塞在 `join` 上，直到所有任务都结束，而 worker 则在后台运行。

```python
    @asyncio.coroutine
    def crawl(self):
        """Run the crawler until all work is done."""
        workers = [asyncio.Task(self.work())
                   for _ in range(self.max_tasks)]

        # When all work is done, exit.
        yield from self.q.join()
        for w in workers:
            w.cancel()
```

如果这些 worker 是线程，我们未必会愿意一开始就全部启动。为了避免在线程还不确定是否必要时就创建昂贵的线程，线程池一般会按需增长。但协程很轻量，所以我们直接把允许的最大数量都启动起来。

这里还有一个值得注意的点：爬虫是如何关闭的。当 `join` 这个 future 完成时，worker 任务虽然还活着，但已经挂起：它们都在等更多 URL，而这时已经没有新 URL 了。所以主协程在退出前会主动取消这些 worker。否则，当 Python 解释器关闭、开始调用各对象析构器时，这些还活着的任务会发出哀嚎：

```
ERROR:asyncio:Task was destroyed but it is pending!
```

那么 `cancel` 又是怎么工作的？生成器还有一个我们之前没展示过的能力：你可以从外部往生成器里**抛入**一个异常：

```python
>>> gen = gen_fn()
>>> gen.send(None)  # Start the generator as usual.
1
>>> gen.throw(Exception('error'))
Traceback (most recent call last):
  File "<input>", line 3, in <module>
  File "<input>", line 2, in gen_fn
Exception: error
```

生成器会被 `throw` 恢复执行，但它现在是在抛出一个异常。如果生成器调用栈里没有代码捕获这个异常，它就会一路冒泡回到顶层。因此，要取消某个 task 所驱动的协程，可以这样写：

```python
    # Method of Task class.
    def cancel(self):
        self.coro.throw(CancelledError)
```

无论生成器当时暂停在某个什么位置，只要是在某个 `yield from` 语句上，它都会恢复执行并抛出这个异常。我们可以在 task 的 `step` 方法中处理这种取消：

```python
    # Method of Task class.
    def step(self, future):
        try:
            next_future = self.coro.send(future.result)
        except CancelledError:
            self.cancelled = True
            return
        except StopIteration:
            return

        next_future.add_done_callback(self.step)
```

这样 task 就知道自己已经被取消了，因此在它被销毁时，就不会再“对着熄灭的光怒吼”。

当 `crawl` 取消完所有 worker 后，它自己也结束了。事件循环看到这个协程已经完成（稍后我们会解释它是怎么知道的），于是它也跟着退出：

```python
loop.run_until_complete(crawler.crawl())
```

`crawl` 方法就是主协程需要完成的全部工作。真正负责从队列中取 URL、抓取页面、解析出新链接的，是 worker 协程。每个 worker 都独立运行 `work` 协程：

```python
    @asyncio.coroutine
    def work(self):
        while True:
            url, max_redirect = yield from self.q.get()

            # Download page and add new links to self.q.
            yield from self.fetch(url, max_redirect)
            self.q.task_done()
```

Python 看到这段代码里含有 `yield from` 语句，就会把它编译成生成器函数。因此，在 `crawl` 中主协程调用 `self.work` 十次时，它并不会立刻执行这个方法，而只是创建出十个带着同一段代码引用的生成器对象。然后再把它们分别包进 Task。Task 接收这些生成器 `yield` 出来的 future，并在 future 完成时通过 `send` 把结果送回生成器。由于每个生成器都有自己的栈帧，它们可以各自独立运行，拥有独立的局部变量和指令指针。

worker 之间通过队列进行协作。它等待新 URL 的代码是：

```python
    url, max_redirect = yield from self.q.get()
```

队列的 `get` 方法本身也是一个协程：它会暂停，直到有人往队列里放入项目，然后恢复并返回这个项目。

顺便一提，这也正是爬虫结束时 worker 暂停的位置：当主协程取消它时，这个 worker 最后一次循环实际上就是在 `yield from` 抛出 `CancelledError` 时结束的。

当一个 worker 抓取某个页面时，它会解析出页面里的链接，把新链接放入队列，然后调用 `task_done` 来减少计数。最终，总会有某个 worker 抓到一个页面，而这个页面上的 URL 都已经抓取过了，同时队列里也没有剩余工作。于是，这个 worker 的 `task_done` 调用就会把计数减到 0。这时，正在等待队列 `join` 方法的 `crawl` 就会被唤醒，并完成执行。

前面我们说过，队列里的项目是这样的二元组：

```python
# URL to fetch, and the number of redirects left.
('http://xkcd.com/353', 10)
```

新 URL 一开始拥有 10 次重定向额度。抓取这个特定 URL 后，服务器会把它重定向到一个带有尾部斜杠的新地址。于是我们把剩余可跟随重定向次数减 1，并把新位置放回队列：

```python
# URL with a trailing slash. Nine redirects left.
('http://xkcd.com/353/', 9)
```

我们使用的 `aiohttp` 默认会自动跟随重定向，并直接把最终响应交给我们。但在这里，我们故意不让它自动跟随，而是让爬虫自己处理重定向。这样，爬虫就能把不同路径上通向同一最终目标的重定向合并起来：如果某个 URL 已经出现在 `self.seen_urls` 里，就说明我们已经从另一个入口开始处理这条路径了。

爬虫抓取 `"foo"` 后发现它会重定向到 `"baz"`，于是它把 `"baz"` 加入队列，同时也加入 `seen_urls`。如果接下来抓取的是 `"bar"`，而它同样重定向到 `"baz"`，那么 fetcher 就不会再次把 `"baz"` 放入队列了。如果响应是一个普通页面而不是重定向，`fetch` 就会解析其中的链接，并把新的链接放入队列。

```python
    @asyncio.coroutine
    def fetch(self, url, max_redirect):
        # Handle redirects ourselves.
        response = yield from self.session.get(
            url, allow_redirects=False)

        try:
            if is_redirect(response):
                if max_redirect > 0:
                    next_url = response.headers['location']
                    if next_url in self.seen_urls:
                        # We have been down this path before.
                        return

                    # Remember we have seen this URL.
                    self.seen_urls.add(next_url)

                    # Follow the redirect. One less redirect remains.
                    self.q.put_nowait((next_url, max_redirect - 1))
            else:
                links = yield from self.parse_links(response)
                # Python set-logic:
                for link in links.difference(self.seen_urls):
                    self.q.put_nowait((link, self.max_redirect))
                self.seen_urls.update(links)
        finally:
            # Return connection to pool.
            yield from response.release()
```

如果这是多线程代码，它将充满竞态条件。举个例子，worker 会先检查某个链接是否已经在 `seen_urls` 里；如果不在，它就把该链接放进队列，并把它加入 `seen_urls`。如果线程恰好在这两步之间被打断，那么另一个 worker 就可能从另一个页面上解析出同一个链接，也看到它“不在 `seen_urls` 中”，于是也把它加入队列。这样，同一个链接就会在队列里出现两次，最轻微的后果也是重复工作和错误统计。

然而，协程只会在 `yield from` 语句处暴露给中断。这正是协程代码比多线程代码更不容易产生竞态的关键区别：多线程代码必须显式地进入临界区、主动拿锁，否则随时可能被打断；而 Python 协程默认是不可中断的，只有在它显式 `yield` 出控制权时才会发生切换。

我们也不再需要回调版本里那个 fetcher 类了。那个类本来就是为了弥补回调的缺陷而设计的：回调在等待 I/O 时，需要一个地方保存状态，因为局部变量不会在多次调用之间自动保留。而 `fetch` 协程和普通函数一样，可以直接把状态存在局部变量里，因此不再需要专门的类来充当状态容器。

当 `fetch` 处理完服务器响应后，它会返回给调用者 `work`。`work` 会对队列调用 `task_done`，然后继续从队列中取下一个 URL 进行抓取。

当 `fetch` 往队列里放入新链接时，它会增加“未完成任务数”，从而让正在等待 `q.join` 的主协程继续保持暂停。相反，如果一个页面里没有新的未见链接，并且它又恰好是队列中的最后一个 URL，那么当 `work` 调用 `task_done` 时，未完成任务数就会降到 0。这个事件会唤醒 `join`，主协程也就完成了。

协调 worker 与主协程的那段队列代码大致如下（见注 13）：

```python
class Queue:
    def __init__(self):
        self._join_future = Future()
        self._unfinished_tasks = 0
        # ... other initialization ...

    def put_nowait(self, item):
        self._unfinished_tasks += 1
        # ... store the item ...

    def task_done(self):
        self._unfinished_tasks -= 1
        if self._unfinished_tasks == 0:
            self._join_future.set_result(None)

    @asyncio.coroutine
    def join(self):
        if self._unfinished_tasks > 0:
            yield from self._join_future
```

主协程 `crawl` 会 `yield from join`。因此，当最后一个 worker 把未完成任务数减到 0 时，它就会发出信号，让 `crawl` 恢复执行并完成。

整个旅程快结束了。程序最开始是通过调用 `crawl` 启动的：

```python
loop.run_until_complete(self.crawler.crawl())
```

那程序又是怎样结束的呢？由于 `crawl` 是生成器函数，调用它时返回的是一个生成器。为了驱动这个生成器，asyncio 会先把它包进一个 task：

```python
class EventLoop:
    def run_until_complete(self, coro):
        """Run until the coroutine is done."""
        task = Task(coro)
        task.add_done_callback(stop_callback)
        try:
            self.run_forever()
        except StopError:
            pass

class StopError(BaseException):
    """Raised to stop the event loop."""

def stop_callback(future):
    raise StopError
```

当这个 task 完成时，它会触发 `StopError`，事件循环把它当作“正常结束”的信号。

不过等等，task 居然也有 `add_done_callback` 和 `result` 这样的方法？你可能已经隐约感觉到，task 看起来很像 future。你的直觉完全正确。我们之前故意对你隐瞒了 Task 类的一个细节：**task 本身就是 future。**

```python
class Task(Future):
    """A coroutine wrapped in a Future."""
```

通常，一个 future 是由别人调用 `set_result` 来完成的；而 task 则在它内部的协程结束时**自己完成自己**。还记得我们之前讨论 Python 生成器时提到过的事实吗？当生成器 `return` 时，它会抛出特殊的 `StopIteration` 异常：

```python
    # Method of class Task.
    def step(self, future):
        try:
            next_future = self.coro.send(future.result)
        except CancelledError:
            self.cancelled = True
            return
        except StopIteration as exc:

            # Task resolves itself with coro's return
            # value.
            self.set_result(exc.value)
            return

        next_future.add_done_callback(self.step)
```

因此，当事件循环调用 `task.add_done_callback(stop_callback)` 时，它其实是在准备“由 task 亲自让循环停下来”。再看一遍 `run_until_complete`：

```python
    # Method of event loop.
    def run_until_complete(self, coro):
        task = Task(coro)
        task.add_done_callback(stop_callback)
        try:
            self.run_forever()
        except StopError:
            pass
```

当 task 捕获到 `StopIteration` 并完成自己时，这个回调就会在循环内部抛出 `StopError`。事件循环因此停止，调用栈也一路展开回到 `run_until_complete`。程序到此结束。

## 结论

如今，越来越多的现代程序受 I/O 限制，而不是受 CPU 限制。对于这种程序来说，Python 线程往往是“两头不讨好”：全局解释器锁阻止它们真正并行执行计算，而抢占式切换又让它们更容易出现竞态问题。在很多场景下，异步才是合适的模式。但随着基于回调的异步代码不断膨胀，它往往会变成一团凌乱不堪的东西。协程则是一种整洁得多的替代方案。它可以自然地拆分成子程序，也拥有正常的异常处理和栈追踪。

如果你把那些 `yield from` 语句看得模糊一些，协程看起来几乎就像一个在线程里执行传统阻塞 I/O 的函数。我们甚至还能用多线程编程中的经典模式来协调协程，根本不需要重新发明一套全新的思路。因此，对于已经熟悉多线程的程序员来说，协程比回调更容易接受，也更自然。

但当我们重新聚焦到 `yield from` 语句上时，就会看到它们清楚地标出了“协程在这里交出控制权，让其他协程运行”。与线程不同，协程明确展示了代码在哪些位置可能被打断，哪些位置不会被打断。Glyph Lefkowitz 在他的文章《Unyielding》中写道：“线程让局部推理变得困难，而局部推理也许是软件开发中最重要的事情。”（见注 14）而显式地让出控制权，则让我们能够“通过检查某个例程本身，而不是检查整个系统，来理解这个例程的行为（以及因此得到的正确性）”。

本章写作时，Python 与异步正处在一场复兴之中。你刚刚学习的这种基于生成器的协程，随着 Python 3.4 中的 asyncio 模块在 2014 年 3 月一同发布。到了 2015 年 9 月，Python 3.5 又把协程直接内建进了语言本身。这些原生协程通过新的 `async def` 语法声明，不再使用 `yield from`，而是使用新的 `await` 关键字来委托给协程或等待一个 Future。

尽管这些细节在演进，核心思想并没有改变。Python 新的原生协程虽然在语法上与生成器区分开来，但工作方式非常相似；事实上，它们在 Python 解释器内部也共享着很多实现机制。Task、Future 与事件循环，在 asyncio 中仍然扮演着各自的角色。

现在你已经知道 asyncio 协程是如何运作的了，反而可以把很多细节放心忘掉。那些复杂机械结构都被藏在一个干净利落的接口之后。但正是对这些基础原理的理解，能让你在现代异步环境中写出既正确又高效的代码。
